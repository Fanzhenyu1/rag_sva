# AVS_AES (AES Core with Avalon-MM Interface)

## 1. Introduction
The AVS_AES design is a complete AES (Advanced Encryption Standard) cryptographic core supporting key lengths of 128, 192, and 256 bits. It implements both encryption and decryption functionality in Electronic Code Book (ECB) mode. The design features an Avalon Memory-Mapped (MM) slave interface for integration into System-on-Chip (SoC) environments, making it suitable for use with processors like Nios II.

Key features include:
- Support for AES-128, AES-192, and AES-256 encryption/decryption
- Configurable decryption datapath (can be disabled for encryption-only applications)
- Avalon-MM slave interface with interrupt capability
- Hardware key expansion with round key storage
- Pipelined datapath for both encryption and decryption operations
- Synchronous design with single clock domain

The core implements the full AES algorithm including:
- AddRoundKey operation
- SubBytes transformation (using S-Boxes)
- ShiftRows transformation  
- MixColumns transformation
- Key expansion for round key generation

|---------------|-------------|--------|

| 0x10-0x13 | Result data block | Read-only |
| 0x14-0x1E | Reserved | - |
| 0x1F | Control/Status register | Read/Write |

### Control Register (0x1F)
| Bit | Name | Description |
|-----|------|-------------|
| 7 | KEY_VALID | Key is valid and expansion can start |
| 6 | IRQ_ENA | Interrupt enable |
| 5-2 | Reserved | - |
| 1 | DEC | Decryption mode enable |
| 0 | ENC | Encryption mode enable |

## 4. Key Logical Behaviors / Sequential Behaviors

### Key Expansion Process
The key expansion module (keyexpansionV2) generates all round keys from the initial user key:
1. User key words are written to shift registers via the Avalon interface
2. When key_stable is asserted, key expansion begins
3. The expansion uses a combination of RotWord, SubWord, and XOR with Rcon values
4. For 256-bit keys, special handling is required every 4th word
5. Expanded keys are stored in internal RAM for later use
6. The keyexp_done signal indicates completion

### Encryption FSM (AES_FSM_ENCRYPT)
States: WAIT_KEY → WAIT_DATA → INITIAL_ROUND → DO_ROUND → FINAL_ROUND

Behavior:
1. Waits for valid key (key_ready) and data (data_stable)
2. Initial round: AddRoundKey with initial key
3. Main rounds (9, 11, or 13 rounds depending on key size):
   - SubBytes → ShiftRows → MixColumns → AddRoundKey
4. Final round (no MixColumns):
   - SubBytes → ShiftRows → AddRoundKey
5. Asserts finished_enc when complete

### Decryption FSM (AES_FSM_DECRYPT)  
States: WAIT_KEY → WAIT_DATA → INITIAL_ROUND → DO_ROUND → FINAL_ROUND

Behavior:
1. Waits for valid key (key_ready) and data (data_stable)
2. Processes rounds in reverse order compared to encryption
3. Uses inverse transformations: InvShiftRows, InvSubBytes, InvMixColumns
4. Asserts finished_dec when complete

### Round Processing
The core uses a pipelined approach where each round takes one clock cycle:
- AddRoundKey: XOR state with round key
- SubBytes: Byte substitution using S-Box ROMs (dual-port for efficiency)
- ShiftRows: Row shifting operation
- MixColumns: Column mixing using Galois Field multiplication

### Avalon-MM Interface Protocol
1. Key setup: Write key words to addresses 0x00-0x07, set KEY_VALID bit
2. Data input: Write data block to addresses 0x08-0x0B
3. Operation start: Set ENC or DEC bit in control register
4. Operation complete: Poll finished status or wait for interrupt
5. Result read: Read result from addresses 0x10-0x13

### Interrupt Handling
- IRQ is asserted when operation completes if IRQ_ENA is set
- IRQ is cleared by any read operation from the result area
- Provides alternative to polling the finished status

### Timing Characteristics
- Key expansion: 81-115 cycles depending on key size
- Data processing: 11-15 cycles depending on key size
- Total latency: 92-130 cycles from data_stable to finished
- Maximum throughput: ~128 bits every 15 cycles at maximum frequency
