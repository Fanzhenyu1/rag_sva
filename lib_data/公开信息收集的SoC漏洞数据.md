# 副本公开信息收集的SoC脆弱性及检测验证方法-不断更新

## P1：安全启动时非法访问
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：在安全启动结束前进行非法访问，可能造成机密性损失
- 涉及的安全资产：bootrom, bios
- 安全属性/策略/规则：在安全启动期间，禁止任何访问

## P2：访问控制时非法访问
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：在访问控制完成前非法访问资产
- 涉及的安全资产：所有关键信息、寄存器
- 安全属性/策略/规则：在访问控制期间，不得发生对资产的非法访问，如HALT操作时，PC值不能被修改和访问;HALT操作时，通过DAP对privilege memory的访问请求应被拒绝

## P3：权限分配错误
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：处理器从控制/状态寄存器（CSR）错误地分配了执行权限级别，导致非特权的用户空间代码可以写入特权的CSR(privileged CSR)寄存器
- 涉及的安全资产：核的状态和特权级
- 安全属性/策略/规则：核的状态和特权级应当正确分配

## P4：未授权的寄存器访问
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：未正确分配或处理寄存器权限，导致权限升级，发生未经授权访问寄存器，如由于代码缺陷（缺少完整case语句等），导致不可预知的越权行为
- 涉及的安全资产：寄存器
- 安全属性/策略/规则：每个关键控制和状态寄存器都应分配并维护正确的寄存器权限级别,只有当处于privilege mode时，CPSR才能被/从其他通用目标寄存器中拷贝

## P5：权限变化漏洞
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：权限级别的设计不当，导致在执行一条指令的过程中变换级别，获取机密信息
- 涉及的安全资产：寄存器
- 安全属性/策略/规则：权限级别在一条指令的执行过程中不能被修改，控制和数据寄存器的读写使能也不能在权限变化时进行更新

## P6：未授权的页面访问
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：未经授权的页面（MMU内存页面）访问请求
- 涉及的安全资产：MMU中寄存器
- 安全属性/策略/规则：MMU内的内存页面访问应始终保持正确的权限级别，如果内存页不允许用户级别的访问，则不应授权给任何低级别的用户内存页的访问权限

## P7：指令寄存器权限问题
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：在中断处理时，不恰当的权限级别设置，可能导致机密性损失
- 涉及的安全资产：Privilege级别
- 安全属性/策略/规则：在中断处理过程中，处理器核的权限级别应始终为机器模式（最高权限）

## P8：PMP地址不匹配
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：PMP单元处理物理内存位置的地址转换，如地址匹配，则会为任何特权模式启用内存访问。如果地址不匹配，则仅在计算机模式下提供访问权限。如果存在调整漏洞，则可以在任何特权模式下提供访问权限
- 涉及的安全资产：PMP中Privilege级别
- 安全属性/策略/规则：如果PMP不匹配，则只能在机器模式下进行内存访问

## P9：内核复位后权限问题
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：复位后内核的权限级别不是机器模式，可能会造成机密性损失
- 涉及的安全资产：Privilege级别
- 安全属性/策略/规则：复位后内核的权限级别必须处于机器模式

## P10：SUM权限漏洞
- 类型：设计缺陷/硬件木马
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：supervisor user memory (SUM)的漏洞，可能导致不当获得了特权级别
- 涉及的安全资产：CSR的SUM中Privilege级别
- 安全属性/策略/规则：存在SUM位（允许SUM访问位）可修改S模式加载、存储和指令获取访问虚拟内存的权限。当SUM=0时，S模式内存访问试图访问U模式可访问的页面应当被禁止。当SUM=1时，允许这些访问

## P11：缓存刷新漏洞
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：缓存刷新指令的设置漏洞，可能导致获得管理员权限
- 涉及的安全资产：Trap Vector Memory (TVM) bit
- 安全属性/策略/规则：当TVM=1时，SFENCE（缓存刷新）指令如果可以在管理员模式下执行，这意味着任何来宾操作系统都可以在软件级别执行缓存刷新。因此当SFENCE指令不支持或预期S模式时，TVM应始终为零

## P12：权限机制漏洞
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：特权级别的操作机制存在漏洞，如PMP模块中的protection bit设置不当错误影响权限，进而影响机密性和完整性
- 涉及的安全资产：Privilege级别
- 安全属性/策略/规则：进程中的最高权限指令不能在低权限级别中执行；任何操作不可从低特权级别升到高特权级别。当目前权限级别低于前一个权限级别时load和store module中的权限不能被修改

## P13：调试相关权限机制漏洞
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：特权级别的设计不当，可导致利用调试模式切换获得特权，获取机密信息
- 涉及的安全资产：debug的Privilege级别
- 安全属性/策略/规则：当从debug模式返回时，必须为重新存储为最高的特权级别

## P14：IP集成漏洞
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：AXI具有不同的security bit，设计不当可导致访问异常
- 涉及的安全资产：AXI security bit
- 安全属性/策略/规则：AXI的security bit设置应与对应IP的安全级别对应

## P15：意外代理或中介
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：该缺陷通常出现在受害者和攻击者都可以访问的共享部件上的转发行为，例如Cache、各种Cache缓冲buffer等，直接非法访问属于前述基础权限错误类别
- 涉及的安全资产：关键信息
- 安全属性/策略/规则：处理器中可跨安全域共享的功能单元、存储单元，需要履行自己在信息流转过程中的安全职责，避免受害者秘密信息经过自己被跨安全域转发

## P16：访问控制粒度不足
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：主要针对设置了隔离策略的跨安全域安全性和寄存器的访问控制，这部分不包括正常权限功能涵盖的部分
- 涉及的安全资产：TEE、多核或者多安全域系统；内存映射寄存器、控制寄存器如MSR、其它寄存器如分支预测寄存器（BTB、BHB、RSB等）
- 安全属性/策略/规则：该安全属性需要保证在跨域共享的资源上不会发生跨安全域的非法信息访问，常发生在共享寄存器、存储、总线等。而且任何寄存器都应该确认是否需要访问控制，一般共享的寄存器都需要确认访问控制安全性。确定不需要共享的私有寄存器则需要验证物理隔离属性

## P17：缺少对备用硬件接口的保护
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：如对镜像或别名内存区域应用了不正确的访问控制，或者对片上结构防火墙中的镜像区域缺少保护，可能导致机密性损失
- 涉及的安全资产：硬件接口
- 安全属性/策略/规则：对备用的硬件接口应有足够的保护机制

## P18：缺少对片上结构或总线安全特性的支持
- 类型：设计缺陷/硬件木马
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：如桥结构中缺少安全检查，可能导致机密性损失
- 涉及的安全资产：桥结构与总线
- 安全属性/策略/规则：桥结构和总线中应有必要的安全检查步骤

## P19：未能禁用保留位
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：存在开发过程中未定义而使用，或者未用的寄存器位没有禁用的情况
- 涉及的安全资产：控制寄存器、状态寄存器、其它私有、通用寄存器
- 安全属性/策略/规则：对于开发过程中未定义而使用、或者保留未用的寄存器位，应该采用禁用机制，防止造成非法访问

## P20：对一次写入位字段的不正确限制
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：对一次性写入位的字段存在不正确的实现方式，比如寄存器设计为在某些值的情况下为写保护，但实际上可能变为只写了一次该值，而不是无论任何值都只能写一次
- 涉及的安全资产：寄存器
- 安全属性/策略/规则：应检查所有一次性写入位的实现方式，确保为无论任何值，该寄存器位都只能写一次

## P21：Fabric网桥安全属性转换不当
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：对于桥结构和总线等，安全属性的变换不当可能导致权限的滥用，造成机密性的损失
- 涉及的安全资产：桥结构与总线
- 安全属性/策略/规则：桥结构和总线中应当设置合适的安全属性转换方式

## P22：Fabric端点总线控制能力设置不当
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：对于桥结构和总线等，其控制能力设置不当可能导致访问控制错误，造成机密性损失
- 涉及的安全资产：桥结构与总线
- 安全属性/策略/规则：桥结构和总线中应设置合适的控制能力

## P23：对暴露的调试和测试接口缺少授权
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：没有实现或对调试接口设置了不正确的访问控制，导致攻击者能够训练调试接口，或者在在某些特定模式（如halt）下，获得对部分内部寄存器（如PC、debug register通常不应被暴露）的访问权限。以及在调试中，将储存安全敏感值的熔丝暴露，泄露信息
- 涉及的安全资产：熔丝
- 安全属性/策略/规则：应保证在调试期间,控制和状态寄存器必须为只读，且不允许对存储器进行读访问；熔丝不应当暴露

## P24：DMA访问漏洞
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：DMA由于设计缺陷，导致访问越界，或者可访问受保护的内存区域，造成机密性损失
- 涉及的安全资产：DMA
- 安全属性/策略/规则：DMA控制器的每次读取和传输必须在规定地址范围内，不能访问受保护的内存区域

## P25：总线的控制FSM漏洞
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：如果在underling logic仍在处理未完成的事务时, memory bus生成错误信号, 则该模块将无条件地认为下一个要处理的信号是可操作的,攻击者可以通过在恶意非法访问之前强制进行错误访问, 从而对任意特权存储器发出读写指令
- 涉及的安全资产：流水线
- 安全属性/策略/规则：在underling logic仍在处理未完成的事务时, memory bus不能为可访问状态，同时也要检查对于错误访问的操作处理

## P26：代码注入漏洞
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：由于设计缺陷，导致可对关键寄存器文件允许代码注入（如内存映射寄存器memory-mapped rigister file）
- 涉及的安全资产：寄存器
- 安全属性/策略/规则：任何关键寄存器和关键信息位置不允许代码注入

## P27：模式位悬空漏洞
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：core中supervisor mode signal处于悬空状态, 从而无法使用SMAP
- 涉及的安全资产：模式位寄存器
- 安全属性/策略/规则：所有的高级模式信号不能悬空

## P28：指令多重访问控制漏洞
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Permissions, privileges, access control
- 漏洞行为描述：LSU的多重访问控制指令设置不当，会最终被LSU接受，而造成机密性损失
- 涉及的安全资产：指令执行流
- 安全属性/策略/规则：当出现多于LSU可控制数目的未完成事务时，LSU应拒绝访问请求

## B1：缓存地址范围漏洞
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Buffer errors
- 漏洞行为描述：总线中的L2 cache地址设置超过特定范围
- 涉及的安全资产：总线地址范围
- 安全属性/策略/规则：所有设置的地址范围均保持在合理区域内

## B2：GPIO可读取cache漏洞
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Buffer errors
- 漏洞行为描述：GPIO能够读写指令cache和数据cache
- 涉及的安全资产：cache读写权限
- 安全属性/策略/规则：外设接口应不能直接读取指令和数据cache

## I1：非法或未知的指令执行
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Information leakage
- 漏洞行为描述：如果非法指令的处理过程存在漏洞，可能导致CSR的数据泄露，或者不确定性
- 涉及的安全资产：CSR内容
- 安全属性/策略/规则：在处理非法指令时，应始终禁止对CSR的访问，以及risc-v中，当执行imm指令时decoder需忽略rs1, rd1域

## I2：地址范围重叠
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Information leakage
- 漏洞行为描述：外设地址范围重叠，可能与高权限的访问区域重叠，如DMA与SPI master接口地址重叠，导致权限升级；或者地址设置错误导致memory aliasing；可能导致不同级别资源的共享内存访问空间，造成信息泄露的问题
- 涉及的安全资产：关键内存空间
- 安全属性/策略/规则：为每个外设定义的地址范围应该是固定的且正确的，且受保护的范围不重叠

## I3：内存越界读取
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Information leakage
- 漏洞行为描述：Dromajo Ram的越界读取,地址读写误操作，可能导致机密信息泄露
- 涉及的安全资产：关键内存空间
- 安全属性/策略/规则：敏感的内存位置不能进行越界写入/读取

## I4：关键信息存储不当
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Information leakage
- 漏洞行为描述：加密密钥存储在未受保护的存储器中，或者通过输出接口泄露（在某些罕见条件触发下）
- 涉及的安全资产：密钥存储
- 安全属性/策略/规则：加密信息需要存储在写保护的内存中，或者使用相应的锁控制来保护，且相关密钥不能通过输出接口泄露

## I5：密码模块复位问题
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Information leakage
- 漏洞行为描述：密码的pass_check位复位阶段未清除，可能造成机密信息泄露，或者该密码被错误地设计为在复位时被hard-coded
- 涉及的安全资产：可传输敏感信息的接口
- 安全属性/策略/规则：每次调试（JTAG）都应有一个密码，并且密码的pass_check位应在复位期间清除

## I6：密码模块时间侧信道漏洞
- 类型：设计缺陷/硬件木马
- 对应脆弱性/漏洞类型：Information leakage
- 漏洞行为描述：在罕见条件触发下（如明文字节）木马导致AES等加密核停止工作，如超时不输出任何值，或者可能由于停止工作导致明文和密文一致
- 涉及的安全资产：密码模块
- 安全属性/策略/规则：加密核应该在规定时间内完成操作（包括轮数），并声明DONE标志位；且任何时刻都不应出现明文和密文相同的情况;加密的输出结果不应和加密模式存在数据路径（可通过侧信道观察到的差异）

## I7：密码模块泄露漏洞
- 类型：硬件木马
- 对应脆弱性/漏洞类型：Information leakage
- 漏洞行为描述：当内部计数器达到阈值时密钥夹在密文中输出
- 涉及的安全资产：密码模块
- 安全属性/策略/规则：任何情况下，密文和明文/密钥都不能存在2bit以上的内容一致

## I8：状态机设计缺陷
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Information leakage
- 漏洞行为描述：状态机设计不当，可能导致机密信息泄露
- 涉及的安全资产：关键状态机
- 安全属性/策略/规则：当两个连续的不受保护状态之间发生状态转换时，应至少固定一位编码 （MSB），即状态间的汉明距离应为1

## I9：重用前未删除敏感信息
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Information leakage
- 漏洞行为描述：释放前未在资源中清除敏感信息以供重用，比如进入调试模式时，处理器没有完全清除安全敏感值，或者在完成任务前泄露中间过程结果
- 涉及的安全资产：缓存、内存、寄存器、密码模块
- 安全属性/策略/规则：应保证所有的存储敏感信息的资源（缓存、寄存器）等在释放给外部重用或者进入调试模式时，已经经过有效复位彻底完成内容清除；当操作完成标志位和读请求未有效时，输出寄存器不能有任何数据，总线读信号必须为0

## R1：资源不当共享
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Resource management
- 漏洞行为描述：温度模块与加密模块混合使用，可能导致信息泄露或权限问题
- 涉及的安全资产：加密模块
- 安全属性/策略/规则：加密模块不应该与常规模块集成在一起

## N1：敏感信息的非法修改
- 类型：设计缺陷/硬件木马
- 对应脆弱性/漏洞类型：Numeric errors
- 漏洞行为描述：攻击者能够对敏感信息进行修改，比如越权修改存储单元中的信息
- 涉及的安全资产：关键信息，内存
- 安全属性/策略/规则：敏感信息应保存在写保护的内存中，且不得随意修改，内存单元在运行期间不允许发生功能破坏，所执行的操作（例如寻址，读写）必须是合法的，所存储的数据是正确有效的

## N2：不可变数据内容存储在可写内存中
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Numeric errors
- 漏洞行为描述：将不可更改的数据存储在可写的内存中，造成数据错误
- 涉及的安全资产：关键信息、内存
- 安全属性/策略/规则：敏感信息应保存在写保护的内存中，如各种密码必须写在写保护的内存中，否则容易被攻击者修改

## N3：模块完整性漏洞
- 类型：硬件木马
- 对应脆弱性/漏洞类型：Numeric errors
- 漏洞行为描述：模块由于硬件木马，导致完整性损失，比如修改加密的输出结果
- 涉及的安全资产：密码模块
- 安全属性/策略/规则：密码核的输出应和最终IP的输出保持一致

## N4：接口/总线完整性漏洞
- 类型：硬件木马
- 对应脆弱性/漏洞类型：Numeric errors
- 漏洞行为描述：篡改接口（如UART）的输出缓存数据
- 涉及的安全资产：接口、总线
- 安全属性/策略/规则：接口、总线收到和转发的数据必须保持一致

## H1：异常指令报告
- 类型：硬件木马
- 对应脆弱性/漏洞类型：Hardware/SoC
- 漏洞行为描述：硬件木马导致解码器逻辑受损，但未报告异常
- 涉及的安全资产：解码器
- 安全属性/策略/规则：所有超出操作码规范的指令编码都应引发硬件异常，此外，处理器还应该跟踪硬件异常并报告问题

## H2：异常指令处理
- 类型：硬件木马
- 对应脆弱性/漏洞类型：Hardware/SoC
- 漏洞行为描述：硬件木马使得在异常处理次数过多时，使处理器进入休眠模式，则可能导致拒绝服务
- 涉及的安全资产：整个处理器
- 安全属性/策略/规则：具备对处理器核异常处理机制的处置方式，不轻易进入休眠状态

## H3：接口密码保护缺失
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Hardware/SoC
- 漏洞行为描述：JTAG接口没有密码保护
- 涉及的安全资产：调试器
- 安全属性/策略/规则：Debug/JTAG接口需要密码保护且密码检查功能需要检查完整密码

## H4：异常处理（超时）
- 类型：硬件木马
- 对应脆弱性/漏洞类型：Hardware/SoC
- 漏洞行为描述：TW（超时等待）位支持拦截WFI指令。当TW=0时，在S模式下允许WFI指令。当TW=1时，如果WFI在S-mode中执行，并且未在特定于实现的有限时间限制内完成，则WFI指令会导致非法指令陷阱
- 涉及的安全资产：超时等待
- 安全属性/策略/规则：具备对处理器核超时等待机制的处置方式，不进入非法指令陷阱

## H5：安全标识符问题
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Hardware/SoC
- 漏洞行为描述：没有唯一的，不可变的安全标识符
- 涉及的安全资产：整个SoC
- 安全属性/策略/规则：应设置唯一、不可变的安全标识符

## H6：错误的默认参数或配置
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Hardware/SoC
- 漏洞行为描述：寄存器的初始值或者模块的初始参数配置不当,可能造成功能错误或者遗失机密信息
- 涉及的安全资产：关键寄存器
- 安全属性/策略/规则：对于寄存器的初始值、模块的初始参数，需要保证其正确性

## H7：硬件逻辑中的竞争条件
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Hardware/SoC
- 漏洞行为描述：在硬件逻辑设计中存在竞争条件，导致信息的数值不确定或发生改变
- 涉及的安全资产：关键信息
- 安全属性/策略/规则：应保证不出现逻辑竞争

## H8：锁保护寄存器的实现不正确
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Hardware/SoC
- 漏洞行为描述：使用寄存器锁定位保护机制，但不能确保锁定位阻止了可对重要硬件系统配置执行更改的系统寄存器或控件的修改
- 涉及的安全资产：带有锁逻辑的寄存器
- 安全属性/策略/规则：应保证寄存器的锁定保护位对重要的硬件系统配置起到了保护作用，使其不被更改，尤其不能被软件覆盖写入，且复位功能不应该清除锁定控制寄存器的值

## H9：电源状态转换后的不当行为
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Hardware/SoC
- 漏洞行为描述：寄存器锁保护功能设计不当，导致某些受保护的寄存器或锁死位在电源状态变化时变为可编程状态
- 涉及的安全资产：带有锁逻辑的寄存器及关键信息
- 安全属性/策略/规则：应保证寄存器及关键信息在电源状态发生变化时，锁逻辑有效，不变为可编程状态，或者可读取

## H10：硬件内部或调试模式可以绕过锁定保护

- 对应脆弱性/漏洞类型：Hardware/SoC
- 漏洞行为描述：寄存器锁死位保护功能设计漏洞，允许攻击者在锁死位已被内部模式或调试功能设置后修改系统配置
- 涉及的安全资产：带有锁逻辑的寄存器
- 安全属性/策略/规则：应保证寄存器在硬件内部或调试模式下，锁逻辑保持有效

## H11：不正确的有限状态机(FSMs)

- 安全属性/策略/规则：应保证带有熔丝的设计必须选择正确的值

## H13：不正确的比较逻辑粒度
- 类型：设计缺陷
- 对应脆弱性/漏洞类型：Hardware/SoC

- 涉及的安全资产：电路设计
- 安全属性/策略/规则：应保证所有的比较逻辑能够抵御时间侧信道的攻击

- 类型：设计缺陷

- 涉及的安全资产：关键信号
- 安全属性/策略/规则：关键信号的警报方式设计，必须阻止非授权的修改或处理

## H18：外设接口使能信号漏洞

- 对应脆弱性/漏洞类型：Hardware/SoC

- 涉及的安全资产：外设IP
- 安全属性/策略/规则：应检查外设接口的使能端，不应始终为高

- 涉及的安全资产：密码模块
- 安全属性/策略/规则：关键信息操作的模块的输入信息不应被篡改

## H20：电路的功耗耗尽漏洞

- 对应脆弱性/漏洞类型：Hardware/SoC
- 漏洞行为描述：由于设计缺乏过热传感器保护或者硬件木马，导致电路过热，存在功耗可能被耗尽，造成DoS
- 涉及的安全资产：电路设计
- 安全属性/策略/规则：设计中应在关键位置具备过热传感器和过热保护功能，将正常工作功耗限制在合理区间，防止耗尽攻击

- 安全属性/策略/规则：模式输入端口应与控制器输入的模式端口内容始终保持一致

## H22：顺序错误漏洞
